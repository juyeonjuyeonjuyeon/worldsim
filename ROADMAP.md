# WS Forest Weather — 구현 로드맵

## 0. 핵심 원칙 (모든 단계에 적용)

1. **물리 모델 우선 (Physics-first).**
   모든 날씨·하늘·자연 현상은 **근본 물리 모델에서 창발**해야 한다.
   "원인을 넣으면 결과가 나오는" 구조여야 하며, 색·값을 손으로 하나하나
   지정하거나 스크립트로 발생시키면 안 된다. 그렇게 쌓은 것은 한 곳을 고치면
   다른 곳이 깨지는 회귀를 부른다(하늘 색 수동 지정이 그 예).
   - 정확도 스펙트럼: **슈퍼컴 정밀 시뮬 / 단순화 물리(← 목표) / 수동 지정**
   - 일반 PC의 목표는 중간 지대: "단순화했지만 **올바른 정성적 인과를 보존**하는 모델".
   - 완전한 수치예보(슈퍼컴)는 불가능·불필요. 인과와 기후 통계가 맞으면 충분.

2. **사람 눈/귀 검수가 필요한 것은 무인 자동에 넣지 않는다.**
   렌더·셰이더·색·노출·톤매핑·사운드 결과물은 "수치상 맞다"로는 검증이 안 된다.
   화면(또는 귀)으로 확인해야 한다. 자동 루프는 잘못된 결과를 "통과"로 착각한다.
   → 밤 암막 버그가 여러 번 재발한 이유가 바로 이것.

3. **코드 구조·데이터·문서·설계는 무인 진행 가능.**
   단, 매 변경 후 **컴파일(parse) 통과를 반드시 확인**하고, 불확실하면 절반 구현 대신 TODO 스텁.

4. **천문 위치 계산은 검증 완료 — 건드리지 않는다.**
   적위·고도·방위·낮길이·v_world_dir_y 지평선 클립은 통과됨.

5. **모바일이 성능 기준선.** (웹·앱·PC 전부 동등 목표)
   모든 신규 효과·물리 모델은 품질 단계(LOD: 고=PC / 중=앱 / 저=웹)로 만들 수 있게 설계.
   → 일반 PC 전략: **무거운 대규모 시뮬은 거칠게/미리 계산, 보이는 국소는 실시간**.

6. **데이터 구조는 처음부터 구면 위상 전제.**
   렌더는 평면 패치여도, 좌표계·격자는 경도 랩어라운드·극 처리를 염두에 둔다.

## 1. 물리 모델 검수 방법론 (핵심 워크플로우)

> 신규 구현뿐 아니라 **이미 만든 모든 현상**에 이 절차를 적용해, "진짜 물리냐 손으로
> 칠한 거냐"를 가려내고 필요하면 근본 모델로 갈아탄다.

각 현상마다:

1. **분류** — `물리 기반` / `절차적 근사` / `수동 지정` 중 무엇인가?
2. **창발 테스트** — "원인 없이 결과가 나오는가?"
   - 입력(수증기·상승·태양 위치 등)을 빼면 결과(비·구름·색)도 사라지는가? → 물리
   - 입력과 무관하게 결과가 박혀 있는가? → 수동/스크립트
3. **판정** — 수동/스크립트인데 회귀가 잦거나 부정확하면 → **물리 모델로 마이그레이션 대상**.
   (단, 모든 걸 물리로 만들 필요는 없다. 고치기 쉽고 그럴듯하면 근사도 OK.)
4. **마이그레이션** — 근본 물리 모델로 교체. 검증기로 인과 보존 확인.

**검증 위계** (이미 만든 검증기 활용):
`L0 천문(정밀 대조) → L1 일사·계절(공식) → L2 메커니즘(원인→결과) → L3 쾨펜(장기 통계)`

**첫 번째 마이그레이션 사례 — 하늘 색:**
현재 태양 고도별 색 테이블 수동 지정 + brt/scotopic/노출 보정 → 조각조각 쌓여 회귀 반복.
→ **물리 기반 대기 산란(레일리+미)**으로 교체. 태양 위치 하나로 낮·노을·박명·밤이
자동으로 나오게. 밤 가시성은 별도 scotopic 층으로 분리. (설계: `docs/SKY_DESIGN.md`)

## 2. 진행 위계 (왜 이 순서인가)

```
Phase 0  렌더 버그 + 하늘 물리 전환   (사람 눈 필요)
Phase 1  묶음 A: 예쁨 + UI            (현 구조, 빠른 개선)
Phase 2  전 지구 격자 + 지형·지질 기반 (가장 큰 구조 — 설계부터)
Phase 3  기상 시뮬 고도화 + 물리 검수  (격자 위에서)
Phase 4  절차적 사운드
Phase 5  배포 (웹 우선 → 모바일/앱)
─────────── 천문·기상 완료 후 ───────────
Phase 6  지질 심화 + 식물 생태         (지형·기후 토대 위에)
Phase 7  생물 생태                     (식물·기후 토대 위에)
```

**의존 순서가 핵심.** 각 층은 아래 층에 의존한다:
지질·지형 → (그 위) 기후·기상 → (그 위) 식물 → (그 위) 동물.
아래가 물리적으로 맞아야 위가 자연스럽게 창발한다. 위층일수록 더 혼돈적이라
"특정 결과 예측"이 아니라 "그럴듯한 창발"이 목표(기상과 동일 원칙).

## 2.5 현재 단기 작업 큐 (2026-06-30 사용자 검수 기반)  ⚠️ 사람 눈 필요

> 렌더 자가검수(윈도우 모드)는 **Godot 에디터를 닫아야** 동작(에디터가 GPU/컴포지터 점유 시
> 백그라운드 렌더 정체). 검수 워크플로우: 변경→헤드리스 파스→윈도우 렌더 PNG→Read로 판정.

### 검수로 확인된 이슈 (원인 규명 완료)

| # | 증상 | 진단(원인) | 분류 |
|---|------|-----------|------|
| 1 | **구름이 안 보임**("달라진 걸 모르겠음") | 적운·권운 불투명도 = `density=1−exp(−τ/12)`≈0.25 → 25% 반투명, 밝은 하늘에 묻힘. 흐림(OVERCAST)만 하늘셰이더가 통째 회색이라 보임. 구름 평면(뭉게·줄기)은 거의 안 그려짐. | 구름 모델 |
| 2 | **오로라 천정 솔기** | 커튼이 `az=atan(vd.x,−vd.z)` 방위각 기반 → 천정에서 방위각 특이점(구 극점)으로 모든 커튼이 한 점 수렴(방사형 바람개비). 물리 오류. | 오로라 셰이더 |
| 3 | **카메라 따라다니는 지면 그림자** | 그림자 아님. 지형셰이더 `edge_fade`(camera_xz 중심 10~28m 블렌딩) + 60×60 굴곡 디테일면 vs 800×800 평탄 지평면의 노멀/솔기 불연속이 카메라 중심으로 이동. 땅뷰에서 두드러짐. | 지면 셰이더 |
| 4 | **오렌지아워 옅음** | 물리적으로 청정·건조 대기는 옅은 황색이 정상 + Filmic이 밝은 지평선 탈색 + Mie 전방산란 백색광·등방 다중산란 fill이 채도 희석. 색 추가가 아니라 **백색 희석항 축소**가 정답. | 산란 파라미터 |
| 5 | **태양·달 작아 보임** | 각지름 0.53°/0.52°는 동결·정확(50° 화각에서 화면 ~1%라 작게 보임). 디스크 키우면 안 됨 → **글로우/후광**으로 존재감(사람눈 모드). | 글레어(디스크 동결) |
| 6 | **UI 부족** | 테스트 탭에 특수현상 토글/강도 슬라이더 없음(오로라 토글만·KP 고정). 날씨 조절 빈약. 드롭다운이 프로그램 설정 날씨 미반영. | UI |
| 7 | 내가 넣은 구름 계절식 부호반전 | `summer=0.5−0.5·cos((m−7)…)`이 7월→0, 1월→1로 뒤집힘. | 버그 |

> 엔진이 이미 매 프레임 렌더 중 → 새로 만드는 게 아니라 파일로 빼내는 것. 부담 작음.
> 특별한 의미: 찍힌 한 장이 "예쁜 게임 장면"이 아니라 **그 좌표·날짜의 실제 천문/날씨 기록**.
> "감상→공유"의 다리 — SNS 공유=홍보(바이럴), 풍경화 소재, 자작 멍때리기 영상 보관.
- [ ] 기본: 스크린샷 저장 / 영상 녹화 / UI 숨김(깨끗한 화면).
- [ ] 포토 모드: 시간 정지 + 자유 카메라로 구도 잡기, 찍기 전 시간·날씨 연출(이미 제어 가능).
- [ ] 사진 표현: 화각·피사계 심도(배경 흐림)·노출 (기존 노출·톤맵 파이프라인과 연결).
- [ ] 시네마틱: 카메라 경로 녹화(부드러운 추적)·타임랩스 캡처(하루·계절 압축).
- [ ] ※ 기술 참고: 고화질 실시간 녹화는 성능 소모 → OS 녹화 위임 / 또는 프레임 저장 후
      오프라인 합성(고품질·비실시간). 구현 시 선택.

## 5. Phase 2 — 전 지구 격자 + 지형·지질 기반 (설계부터)

> 프로젝트에서 가장 크고 되돌리기 어려운 단계. **코드 전에 종이 위 설계 먼저.**
> 여기서 만드는 지형·지질이 이후 식물·동물의 토대가 된다.

### 5-1. 설계 (무인 가능 — 설계 문서)
- [ ] 축소 행성 크기 / 격자 해상도 / LOD 단계
- [ ] 지형·기후·식생 데이터 출처 (실제 지구 데이터 vs 절차적 생성)
- [ ] 셀 상호작용 여부 (지점 독립 vs 격자 이류) / 경도 랩어라운드·극 처리

### 5-2. 구현
- [ ] 좌표별 기후 데이터 (쾨펜 기후대 매핑)
- [ ] 지형: 산·언덕·강·바다·호수·연못
- [ ] 땅 재질·성분: 흙·진흙·모래·화강암·현무암 등 (기후·지역별)
      ※ 가능하면 풍화·퇴적 같은 단순 물리 규칙으로 분포가 창발하게
- [ ] 좌표별 식생 분포 (기후대 기반 — Phase 6의 씨앗)

### 5-3. 지도 UI + "토양 종류 = 공통 입력" 구조  ★ 핵심 설계

> 목표(마인크래프트 바이옴 류): **지도 UI에서 지역 클릭 → 그 지역의 지형·기후·토양·자생식물·날씨가 세팅.**
> 단 고정 규칙표가 아니라 물리/데이터에서 창발. 좌표 결정 방식 두 갈래:
> 실제 지구 데이터(위경도→쾨펜·고도·지형, 쉽고 정확) vs 절차적 생성(노이즈→창발, 더 복잡).

- [ ] 지도 UI: 지역 클릭 → 좌표 세팅 → 지형·기후·토양·자생식물·날씨 일괄 적용.
- [ ] **토양 종류를 여러 시스템의 공통 입력으로 설계** (이게 의존 순서의 핵심):
      토양 종류 1겹이 정해지면 → 지형 외형 / **투수율(웅덩이)** / 식생(자생식물) / 사운드(빗소리)가
      모두 그것을 받아쓴다. = 지질(토양)이 식물·날씨·소리의 아래층.
- [ ] **토양별 투수율 → 웅덩이 차등** (난이도 중, 일부 구현됨):
      모래=잘 빠짐(웅덩이 적음) / 진흙·화강암=안 빠짐(웅덩이 고임) / 흙=중간.
      ※ 빗물 웅덩이·범람·지면 습윤은 이미 구현됨(감사표) → **투수율 파라미터만 추가 연결**하면
      같은 비에도 토양별로 웅덩이가 다르게 창발. 새로 만드는 게 아니라 입력 한 겹 연결.
- [ ] 토양별 빗소리 차이(모래·진흙·자갈·수면)는 **Phase 4 사운드**에서 처리
      (`AUDIO_DESIGN.md`의 재질별 채널). 입력(토양 종류)은 여기서 정하고 소리는 Phase 4가 받아씀.

## 8b. 구조물(에셋) ↔ 날씨 상호작용  (날씨 완료 후 — 영상·콘텐츠 제작용)

> 용도 예: 한옥에서 비 오는 풍경 멍때리기 영상. 건물은 에셋으로 넣되 이 월드 날씨와 리얼하게 반응.
> 핵심 구조: **"재질 태그 = 공통 입력"** (지질의 토양 종류와 같은 패턴).
> 에셋이라서 상호작용이 막히는 게 아니다 — 날씨 시스템이 에셋 표면에 작용하는 것.
> 전제: **에셋 각 부위에 재질 태그(기와·나무·흙 등)가 있어야** 시스템이 맞게 작용.
> → 에셋 고를 때 "재질별 머티리얼 분리" 여부가 중요. 통짜 에셋은 수동 태깅 필요.

- [ ] **젖음**: 기와·기둥·마루가 비에 색 짙어지고 광택 (지면 습윤과 동일 원리, 표면에 적용).
- [ ] **처마 빗물**: 처마 끝에서 떨어지는 물방울/물줄기 (강수 연동 파티클 — 영상 핵심 정서).
- [ ] **빗물 고임**: 마당 바닥에 기존 웅덩이 시스템 적용 (토양 투수율 연동).
- [ ] **적설**: 지붕·담장·마루 위 향한 면에 눈 쌓임 (지붕 형태 따라 자연스럽게).
- [ ] **바람 반응**: 풍경(風磬)·창호지 문·마당 나무.
- [ ] **채광·그림자**: 처마 각도에 따른 마루 빛, 시간 따라 움직이는 그림자 (태양 계산과 직결 —
      한옥 특유의 "처마 깊이에 따른 빛"이 자동으로).
- [ ] **재질별 소리**: 빗방울이 기와/마루/흙/물받이에 닿는 소리, 발소리 → Phase 4가 재질 태그를 받아씀.
- [ ] (후순위, 난이도 상) 파괴·변형: 창호지 찢김, 눈 무게 처마 휨 등 — 식물 파괴 물리와 동일하게 후순위.

> 새로 만드는 게 아니라, **이미 만든 날씨 상호작용을 재질 태그를 매개로 에셋 표면에 연결**하는 작업.

**세계 문화 유적지로 확장 (8b의 일반화):**
- [ ] 유적지 = **에셋 + 실제 좌표** → 그 위도·고도·계절의 **진짜 날씨·천문**이 자동으로 펼쳐짐.
      예: 마추픽추(남위13°·고도2400m, 서늘·안개) / 앙코르와트(북위13°, 열대 스콜) /
      페트라(북위30°, 건조·쏟아지는 별). **그 장소의 실제 하늘**이 킬러 포인트(예쁜 하늘 갖다붙이기 아님).
- [ ] 좌표·에셋만 바꾸면 각각 다른 멍때리기 콘텐츠. 한옥을 제대로 만들면 유적지는 대부분 그 일반화.
- [ ] 한계: 주변 지형(산·계곡)도 분위기의 절반이라 건물만으론 부족(한옥보다 손 더 감).
      문화적 정확성 유의(자료 보고 제작 — 태도 문제).

## 8c. 경험 모드 사다리 — 영상·라이브·게임화  (장기 목표, 지금은 기록만)

> 같은 월드를 여러 결과물로. **자유도가 한 칸씩 오르는 디딤돌**로 설계 — 각 단계가
> 앞 단계 위에 조금씩만 얹어, 결국 가장 어려운 자유 보행까지 자연스럽게 도달.
> 핵심 원리: 어려운 부분(거대한 걸어다니는 월드)은 **창밖으로 밀어내고**, 플레이어가
> 실제로 사는 공간은 **작고 통제 가능하게** 한정한다.
>
> **탈것은 "안"이 풍경만큼 중요하다.** 플레이어가 실제로 머무는 곳은 그 작은 내부.
> 작게 한정했기에 **디테일을 쏟아부을 수 있다**(넓고 얕은 창밖 ↔ 좁고 깊은 내부).
> - 내부 시각: 창틀·계기판·좌석 등 **프레임**(풍경을 액자처럼), 창으로 든 빛이 내부에 드리움
>   (태양 계산과 직결·한옥 채광과 동일), 창에 맺히는 비/김, 흔들리는 손잡이·장식(생명감).
> - 탈것별 사운드: 기차=레일 덜컹·터널 반향 / 차=엔진·노면 마찰(재질 연동)·지붕 빗소리 /
>   비행기=순항 화이트노이즈. 시점별로 다르게(운전석 vs 짐칸: 엔진 멀어지고 바람·하늘 가까워짐).
> - **재질 태그·공간 잔향 시스템 공유**(8b 한옥과 동일 원리) — 새 시스템 아님, 작은 공간에 정밀 적용.
> - ※ 내부 모델은 **에셋 의존**(한옥처럼). 물리는 그 위 빛·습기·소리·움직임을 입힘.

**난이도 사다리 (쉬움 → 어려움):**
- [ ] **① 한옥 등 고정 풍경** — 카메라 고정, 풍경 안 움직임. 가장 쉬움(8b).
- [ ] **② 기차** — 레일 고정 → 전 지구 불필요, **경로 양옆 풍경 띠만** 생성.
      상호작용은 객차 안(작은 공간)만, 창밖은 무한 절차 생성(앞 생성·뒤 폐기).
      플레이어 충돌·자유이동 문제 없음. **걷기 오픈월드의 저비용 대안.**
- [ ] **②' 차량(트럭·승용차)** — 도로 고정이라 기차와 동일 난이도 칸. 시점이 다양한 게 매력:
      · 조수석/운전석(창밖 풍경) · **짐칸에 누워 하늘 보며 달리기**(천문·구름이 가장 빛나는 뷰 —
        별·구름·노을이 위로 흐름) · 오픈카/창 내린 상태(바람·비 체감).
      · 기차와 같은 구조(경로 리본·앞 생성/뒤 폐기·차내만 상호작용)라 거의 같은 작업으로 추가됨.
- [ ] **③ 스누피 비행기(낮은 공중)** — 곡률 무시 가능, 만화풍이라 디테일 부담↓.
      기차의 공중 버전.
- [ ] **④ 여객기(고고도)** — 지형 디테일 부담↓(멀리서 봄)·**구름 뷰가 강점**(구름 위/뚫고/노을),
      단 지평선 **곡률·넓은 시야 스케일↑**. 구면 위상 전제가 여기서 쓰임.
- [ ] **⑤ 자유 보행** — 가장 어려움. 아무 데나 가므로 결정적 지형·충돌·지속성 전부 필요.

**라이브 방송 형태 (타임랩스보다 쉬움):**
- [ ] 유튜브 24h 라이브식 **무한 산책/주행** — "완주"가 목표가 아니라 "지금 풍경"이 전부.
      한 바퀴 닫을 필요 없음(랩어라운드 불필요), 앞쪽만 생성. **전 지구 격자의 쉬운 진입점.**
- [ ] 위도를 천천히 올리며 기후 전환(열대→온대→설원)이 자연히 흐르게 — lo-fi 정서.

**게임화 (가장 먼 목표 — 정직한 격차):**
- [ ] 영상→게임은 "조금 더"가 아니라 **새 인프라 층**(결정성·충돌·물리·저장·성능 보장·네트워크).
      물리 엔진(날씨·천문·지형)은 **자산으로 재사용**되지만 게임화는 별도 작업.
- [ ] 경로: 풀 오픈월드 직행 금지 → **걸을 수만 있는 작은 인터랙티브 데모**부터 계단식.

**자연 상호작용 샌드박스 (생존 게임 아님 — 도시인의 자연 체험이 목적):**
> 목표·점수·죽음 없이, 자연을 손으로 다루는 경험 자체가 목적. 생존 게임의 무거운 부분
> (체력·허기·게임루프·밸런싱)이 빠져 → 남는 건 **재료 물리**. 단 새 난이도 축이 열림.
- [ ] **재료가 지질·식물에서 실제로 유래** = 진짜 인과(강점). "이 지역 이 광물→이 안료"가
      규칙표가 아니라 실제에 기반. 도시인이 흙·돌·잎을 진짜처럼 만지는 체험.
- [ ] 난이도 — **무른 재료·상태 변화 = 가능**: 점토 도자기(빚기), 암석 갈아 안료, 으깨기(잎→즙).
      voxel/메시 변형·상태 변화로 자연스럽게.
- [ ] 난이도 — **단단한 것 자유 절단·천공 = 가장 어려움**: 나무 임의 베기, 돌 구멍, 잎 임의 절단.
      실시간 메시 절단 필요(미리 만든 모델로 못 함). 무른 재료 다음의 큰 도전.
- [ ] 지질·식물 완성 이후. 게임루프 없어 게임화보다 가볍지만, 재료 변형 물리가 야심참.

**화학 (선 긋기 — 중요):**
- [ ] **자유로운 화학 전반 = 불가**. 임의 물질 조합 결과를 물리로 창발시키려면 분자 단위 →
      일반 PC 밖(슈퍼컴도 완전히는 못 함). "다 할 수 있다"는 성립 안 함.
- [ ] **미리 정한 반응 몇 개 = 가능**(규칙표): 안료 혼합·가열/굽기(도자기)·발효 등,
      자연 체험에 자연스럽게 녹는 것.
- [ ] ※ 화학은 **'물리 창발'이 아니라 '규칙표'** — 프로젝트 철학(진짜 물리)과 결이 다름.
      자연 체험에 녹는 선까지만. 본격 "화학 실험실"화는 마인크래프트식 레시피 게임으로
      **정체성이 분기**하는 결정 — 주의.

**풍경화 모드 (이 프로젝트와 결이 가장 잘 맞는 축):**
> 만든 풍경을 **소재로** 그림. 관찰→표현의 자연스러운 연장(멍때리다 그리기) → 정체성 부합.
> 강점: 사용자 미술(유화) **전문성** + 안료 시스템(돌 갈아 만든 물감)과 연결.
- [ ] **매체별 고유 물리** — "선 긋는 도구" 하나로 뭉뚱그리지 말고 매체마다 물성을 다르게:
      · 연필: **깎아 써야 함(소모)·쓸수록 뭉툭·흑연이 종이 결에 박힘·문지르면 번짐·지우개로 흐려짐.**
      · 목탄: 더 잘 번지고 가루 날림·뭉개기 쉬움.
      · 수채: 물 번짐·종이 젖음·마르며 가장자리 진해짐·투명 중첩.
      · 유화: 젖은 채 밀림·두껍게 쌓임(임파스토)·나이프 자국·천천히 마름·글레이징.
      · 펜/잉크: 번짐 적고 선명·잉크 고임.
- [ ] **물감/매체 물리(섞임·번짐·질감·건조) = 잘 됨.** 사용자 전문성이 빛나는 영역. 감산 혼합(물감답게).
- [ ] **도구 거동 사실성(붓 종류·나이프·연필 깎임 등) = 됨.** 도구마다 물리 모델 분리.
- [ ] **손 촉각 경험 = 입력 장치 의존.** 태블릿(필압·기울기)으로 상당히 / 완전한 촉감(붓 저항·연필
      긁힘)은 햅틱 하드웨어 필요(아직 일반적이지 않음) → 시각·청각으로 보완.
- [ ] **VR 결합 = 정점.** "자연 속(마추픽추 안개 등)에 서서 이젤에 그리기" — 도시인이 못 하는
      '자연에서 풍경화' 체험. VR 컨트롤러가 붓이 되면 도구 경험도 한 단계 진짜에 가까워짐.
- [ ] 한계: 페인팅 엔진(매체 물리)은 그 자체로 별도 큰 작업. "실제 도구 감각"의 끝은 하드웨어 의존.

**경쟁 지형 (차별점 근거):** 전 지구 물리 시뮬(NVIDIA Earth-2 등)=슈퍼컴·못 걸음 /
지구 전략·교육 게임(Imagine Earth 등)=탑다운·걷지 않음 / 워킹 시뮬=1인칭이나 날씨·천문이 가짜(스카이박스).
→ **"일반 PC / 물리로 창발하는 날씨·천문 / 1인칭 산책"의 조합은 빈자리.**
빈 이유는 "아무도 생각 못 해서"가 아니라 **너무 어려워 다들 한 조각씩만** 해서 → 차별점이자 도전.

**아이디어 보관함 (미검토 — 기록만):**
- 바벨탑/수직 등반: 고도 오를수록 환경이 실제 물리로 변함(기온↓·공기 희박·하늘 짙어짐→검어짐·
  낮에도 별). 대기 산란이 고도 입력으로 자동 생성 → 시스템 쇼케이스. 쌓기 메커니즘은 별도.
- 사다리로 태양·달 닿기: 실제 거리(달 38만km·태양 1.5억km)라 "실제 우주 기반"과 모순 → 닿으려면
  스케일 압축 필요. 세 갈래 — (A)완전 실제=안 닿음(닿을 수 없는 경외) / (B)동화 우주=천문 포기 /
  (C)방향·환경은 실제·거리만 압축해 도달(거짓말 최소). 어느 쪽이든 **메인 천문과 분리된 별도 모드**.

### 8d. 행성 확장 — 다른 천체의 하늘·날씨  (장기, 추진 의사 있음)

> 핵심: **같은 산란 엔진 + 행성별 대기 파라미터.** 색을 칠하는 게 아니라 그 행성 대기 물리에서
> 하늘이 창발 — 지구에서 한 것과 동일 철학. (식물의 "공통 엔진+종별 파라미터"와 같은 구조.)
> 큰 구분: **하늘 색·박명·별 = 파라미터 교체로 잘 됨** / **날씨(구름·강수·바람) = 행성마다 고유 물리 새 작업.**

- [ ] **★ 1순위: 화성** — 산란 엔진이 가장 보람차게 확장. 레일리 산란(CO₂ 95%) + **먼지(에어로졸) 항** 추가.
      먼지가 파란빛 흡수 → **황갈색(버터스카치) 낮 하늘**, 그리고 **지는 해 주변이 파란 노을**(지구와 정반대!).
      먼지로 **박명이 길게** 지속. 학계에 화성 대기 색 시뮬(레일리+먼지 산란·흡수) 참고 자료 있음.
      날씨 여지: 먼지폭풍·옅은 CO₂/물얼음 구름(나중 확장).
- [ ] **★ 2순위: 달** — 대기 없음 → 산란 끄면 됨(가장 단순). **낮에도 검은 하늘 + 별.**
      날씨는 없음(대기 필요) → "풍경·천문" 경험: **지구돋이**(지평선 위 지구), 쏟아지는 별.
      화성("낯선 대기")과 대비되는 "대기 없는 우주의 고요".
- [ ] 이후(우선순위 무관): 타이탄(질소+유기 안개=오렌지-갈색 하늘, 메탄 비) /
      금성(황산 구름=누런 하늘, 단 대기 너무 두꺼워 **몬테카를로 산란 필요=가장 무거움**) / 수성(달과 유사).
- [ ] 구조: 메인 지구 천문(검증됨)은 그대로 두고, 행성별 대기 파라미터 세트를 갈아끼우는 모드로.

**우주 여행 (행성 확장의 연장 — 단 스케일이 정체성을 가름):**
> "여행"의 뜻에 따라 난이도 극단적으로 갈림. 핵심 갈림길: **실제 거리를 지키면 못 가고,
> 가려면 실제를 포기**해야 함 = 기술 문제 아닌 **정체성 결정**.
- [ ] **(1) 행성 표면에서 하늘 보기 = 됨** (8d 그 자체). 화성 하늘·달 지구돋이.
- [ ] **(2) 우주 공간에서 천체 감상 = 됨, 멋짐.** 우주선/공간에서 지구·달·태양·별을 **실제 위치에**
      배치하고 카메라를 둠. 경험 사다리(여객기→대기 벗어나 우주)의 끝. 천문 계산이 그대로 빛남.
- [ ] **(3) 행성 간 이동 = 스케일 벽.** 화성도 5천만km+(우주선 몇 달) → 실제 속도론 도착 불가.
      가려면 거리 압축/워프/시간 점프 = **실제 우주와 충돌**(사다리로 달 닿기와 같은 선택).
- [ ] **(4) 항성 간 = 사실상 판타지.** 가장 가까운 별도 4광년. 압축으로도 감당 불가 →
      절차 생성 가짜 은하(No Man's Sky류). "실제 우주 기반" 완전히 상실.
- [ ] ※ **(1)(2)=프로젝트 확장**(실제 자리에서 실제 천체 감상) / **(3)(4)='우주 탐험 게임'으로 정체성 분기.**
      앞은 "실제 우주를 여러 자리에서 감상", 뒤는 "우주를 무대로 한 새 게임"(별도 프로젝트에 가까움).

## 9. Phase 6 — 지질 심화 + 식물 생태  (천문·기상 완료 후)

> 기후·지형 토대 위에서. 식물은 빛·물·기온·토양에서 자라야 한다(창발).

### 9-1. 지질 심화
- [ ] 풍화·침식·퇴적 (단순 물리 규칙) → 지형·토양 분포가 시간에 따라 변화
- [ ] (선택, 오프라인 precompute) 지질 시간 스케일 지각 변동 — 산맥·해안선의 "역사"
- [ ] 토양 성분이 기후·암석·식생과 맞물리게

### 9-2. 식물 생태 — FSPM 기반 (Functional-Structural Plant Model)

> 핵심 접근: 종마다 코드를 새로 짜지 않는다. **공통 생장 엔진 1개 + 종별 파라미터 세트**.
> 감나무·침엽수의 차이 = 잎/가지 에셋을 새로 만드는 게 아니라, 같은 엔진에
> 분기각·생장률·잎 배치 등 **숫자(파라미터)를 다르게 주는 것**. (LIGNUM·L-Py·Helios 방식)

**구현 경로 (수직 슬라이스 우선 — 한 종을 끝까지, 그다음 종 확장):**
- [ ] 공통 L-시스템 생장 엔진 (가지·줄기 3D 구조: 분기각·길이·굵기·발산각)
- [ ] 빛 적응 연결 (그늘→위로, 트인 쪽으로. 이미 만든 태양 위치 계산을 입력으로)
- [ ] 종별 파라미터 세트 (유명 식물부터: 감나무·소나무 …) — 새 종 = 새 파라미터표, 새 코드 아님
- [ ] 계절 변화: 개화·낙엽·생장·단풍 (기온·일조 연동)
- [ ] 식생 분포가 기후·토양·고도·일사에서 창발 (쾨펜 식생대와 일치 검증)
- [ ] LOD: **근경=풀 FSPM 실시간 / 중경=생장 스냅샷 몇 단계 / 원경=분포 통계 칠하기**
- [ ] 생장은 **비실시간 허용** (나무는 몇 달~몇 년 — 백그라운드/오프라인으로 키워 결과만 표시)

**메쉬/에셋 경계 (중요):**
- 알고리즘 자동 생성: 가지·줄기 구조, 잎·꽃·열매의 **위치·각도·개수·배치(엽서)**, 생장, 계절 변화.
- 에셋(또는 별도 절차적) — 종별 1세트만: **잎 한 장·꽃 한 송이·열매 한 알의 단일 윤곽/텍스처**.
- 즉 "나무 통째 에셋"이 아니라 **말단 프리미티브 한 장만 에셋**, 나무 전체는 자란다.

**참고 도구/모델:** L-Py(L-시스템 엔진), OpenAlea(플랫폼), Helios(종 공통 파라미터+충돌물리),
LIGNUM(침엽수·활엽수 공통 4단위 구조). Godot용 완성 플러그인은 없음 → 알고리즘 직접 구현/포팅.

### 9-3. 식물 ↔ 기후 상호작용 (방향별 난이도)

- [ ] **기후 → 식물 (난이도 중, FSPM 핵심 — 잘 됨):**
      빛 따라 가지 생장, 계절 따라 잎/꽃/열매/단풍, 물·기온 부족 시 생장 저하·시들음,
      기후대별 종 분포. 이미 만든 날씨·일사·계절이 그대로 입력.
- [ ] **식물 → 미기후 (난이도 중상, 부분·단순화):**
      그늘(나무 밑 냉각·차광 — 차광은 이미 계산), 방풍, 증산에 의한 국소 냉각·습도.
      숲이 지역 강수를 바꾸는 수준은 격자 기상과 얽혀 무거움 → 국소 효과 근사까지.
- [ ] **식물 ↔ 식물/토양 (난이도 상, 후순위 — Phase 7과 묶임):**
      빛·물 경쟁(큰 나무가 작은 나무를 그늘로), 낙엽→토양 비옥화, 다음 세대 발아.
      = 생태계 시뮬(번식·세대). 개인 PC엔 부담 → LOD·통계로 크게 타협하거나 후순위.

### 9-4. 날씨 ↔ 식물 표면 상호작용 (실시간 — 생장과 별개, 시간 스케일 짧음)

> 9-2 생장은 몇 달~몇 년. 이 항목은 실시간~몇 시간. 분리해서 다룬다.
> 상당수가 이미 지면/고정 나무에 구현됨(감사표) → FSPM 나무로 옮기고 연결하는 작업.

- [ ] **눈 쌓임 (난이도 중):** 가지의 위 향한 면·수평 가지에 적설. FSPM 구조라
      종별로 쌓이는 모양이 자동으로 다름(침엽수 잎뭉치 위 vs 활엽수 가지 위).
      (지면/가지 적설은 이미 구현 — 자라는 나무에 연결.)
- [ ] **서리·이슬 (난이도 중):** 잎·가지 표면에 흰 서리/물방울 셰이더.
      조건(기온<0°/이슬점)은 이미 계산됨 — 표면에 얹기.
- [ ] **젖음 (난이도 중):** 비 맞은 잎·줄기 색 짙어짐·광택 (지면 습윤과 동일 원리).
- [ ] **바람 흔들림 강화 (난이도 중):** 이미 흔들림 있음 → FSPM 가지 구조로
      가지별 차등 흔들림(끝가지 크게/줄기 거의 안 흔들림), 잎 뒤집힘.
- [ ] **(후순위, 난이도 상) 파괴 물리:** 눈·얼음 무게로 가지 처짐/부러짐, 착빙 코팅,
      우박 손상, 바람에 의한 가지 변형. 무거움 → 단순화하거나 후순위.

> 원칙 재확인: 아래층(기후)이 위층(식물)을 미는 방향은 자연스럽고 잘 된다.
> 위(식물)가 아래(기후)로 거슬러 영향 주는 방향은 단순화가 필요하다.

## 10. Phase 7 — 생물 (동물·곤충)  (식물·기후 완료 후)

> 맨 위층. 식물(먹이)·기후·지형이 깔려야 그 위에서 의미가 생긴다(토끼는 풀이 자라야 먹는다).
> **모듈식**으로 설계 — 처음부터 생태계로 깔지 말고, 필요한 영상·씬에 모듈을 끼운다.
> 목적별로 깊이가 다르다: 멍때리기 영상=앰비언트만 / 제대로 된 생물 시뮬=개체 상태 기반까지.
> 핵심 철학(프로젝트 공통): 행동을 애니메이션으로 "칠하지" 말고, 개체 내부 상태에서 **창발**시킨다.

### 10-1. 앰비언트 생물 (난이도 하~중, 영상·배경용 — 1차 목표)
- [ ] 떼 움직임 (Boids: 정렬·응집·분리) — 새 떼·물고기 떼·반딧불이. 수백~수천 마리 실시간 OK.
- [ ] 환경 반응 규칙(가벼운 상태머신): 시간·날씨 입력으로 "언제 어떤 동물이 얼마나 보이나".
      밤→새 사라지고 반딧불이 등장, 비→활동 감소, 추위→둔화. (이미 만든 날씨·시간이 입력.)
- [ ] 원경·실루엣은 단순 모델로(노을에 나는 새 실루엣 등) — 에셋 부담 작음.
→ 멍때리기 영상·애니메이션 배경은 여기까지로 대부분 충족. 식물 생태계 완성 전에도 가능.

### 10-2. 개체 상태 기반 행동 — "실제에 기반한 개체별 자유도" (난이도 중상)

> 떼로 뭉뚱그리지 않고, **개체 하나하나가 자기 상태에 따라 스스로 판단**. 같은 종이라도
> 더 배고픈 개체는 위험을 무릅쓰고, 배부른 개체는 숨는 식의 개체차가 창발.
> 네 겹으로 쌓는다:

- [ ] **(1) 개체별 내부 상태:** 배고픔·갈증·체온·피로·경계심 등. 시간·환경(날씨·기온)에 따라 변화.
      (추우면 체온↓, 시간 지나면 배고픔↑ — 이미 만든 기온·시간이 입력.)
- [ ] **(2) 상태 기반 행동 선택:** 내부 상태가 행동을 고름(배고픔>임계→먹이찾기, 경계심>임계→은신).
      애니메이션을 짜는 게 아니라 상태에서 행동이 나온다.
- [ ] **(3) 개체별 기질 파라미터:** 겁많음·활동성 등이 개체마다 조금씩 다름 → 같은 상황에 다른 반응.
      = "개개인의 특성"(FSPM의 종별 파라미터처럼, 여기선 개체별).
- [ ] **(4) 종별 실제 습성:** 야행성·활동 온도범위·무리 크기 등 실제 생태 데이터를 파라미터화.
      알고리즘 난이도라기보다 **자료 조사량**의 문제(종마다 실제 습성 조사→파라미터 변환).

### 10-3. 정직한 한계 (이 층의 벽)
- [ ] **포식-피식·번식·세대 (난이도 상, "되면 흥미로운" 선택):** 개체군 동역학(Lotka-Volterra류).
      영상엔 불필요, 게임 생태계엔 선택적으로.
- [ ] **진화·적응 (난이도 상, 백그라운드용):** 세대 거듭하며 형질 변화·종 분화 창발. 가능하나
      수백~수천 세대 필요 → 느리고 무거움. 실시간 관람보다 백그라운드.
- [ ] **진짜 학습·인지 (비목표):** 강화학습 수준의 "스스로 배우고 전략 떠올림"은 개인 PC에 비현실적.
      다행히 실제 동물 행동 상당수는 학습이 아닌 본능(상태→반응)이라 10-2로 충분히 표현됨.
- [ ] **몸·애니메이션 (식물보다 어려움):** 식물은 L-시스템으로 형태까지 자동 생성됐지만,
      동물의 골격·보행 애니메이션은 절차적 생성이 훨씬 어렵다 → **몸·기본 동작은 에셋 의존도 높음.**
      자동 생성되는 건 주로 "행동"이지 "몸"이 아니다. (원경·실루엣은 부담 작음.)
- [ ] **규모:** 개체 상태 기반은 떼보다 무겁다 → 근경 몇 개체만 풀 시뮬, 멀리는 떼/통계 LOD.
      멍때리기 영상은 화면에 보이는 수가 적어 오히려 유리.
